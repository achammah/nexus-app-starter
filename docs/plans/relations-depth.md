# Lane: relations-depth — stable relation identity, inverse labels, many-to-many, polymorphic, integrity

IDENTITY MODEL (the architecture call): **store ids, read labels.** Rows persist relation values as target row IDS — single: `"co_1"` · many: `["ce_1","ce_2"]` · polymorphic: `{object,id}` (id prefixes can collide across objects, so poly refs carry their object key). Every READ surface (list/get) projects the display label from the target's primary via the store (fullName-primary → joined name), so the external API keeps returning the SAME label strings it returns today — zero breakage for existing journeys, exports, palette, related-list derivation — and appends `_refs: {fieldKey: id|ids|ref}` for UI that needs identity (picker preselect, jump, disambiguation). WRITES accept an id, or a primary-label string that resolves uniquely (normalized to id at write; two candidates → 400 listing them; no match → kept verbatim as a dangling label, today's behavior). Seed/legacy label values normalize to ids in one idempotent boot pass (pure function of config+rows → replay-safe; runs before the command-log replay; store counter untouched). Consequences: RENAME of a target needs no sweep — labels re-project live (the root fix for carry-in a); MERGE re-points ids (arrays deduped, poly by object+id); TRASH keeps ids (label still projects via getAny; restore heals silently); DESTROY severs inbound refs inside the already-logged destroyRow (null / removed from arrays). Search `q` haystack includes projected labels.
