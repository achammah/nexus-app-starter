# Lane: schema-editor — runtime schema editing (config-as-data: seed + logged deltas)

DELTA REPRESENTATION (the architecture call): **the command log IS the delta store — no parallel structure.** Three new LOGGED store ops mutate the store's in-memory config IN PLACE: `schemaObjectAdd(def)` · `schemaFieldAdd(objKey, fieldDef)` · `schemaFieldUpdate(objKey, fieldKey, patch)` (retire/re-activate = `{isActive}` through Update — no separate op). `starter.config.json` stays the immutable SEED; boot = parse seed → seed rows → replay, and each replayed schema op re-applies its mutation at its original seq, so schema deltas and the data writes that depend on them interleave in strict log order (a field is always defined before the patch that fills it). MERGE POINT: none needed — the server constructs `new Store(CONFIG)` with the SAME object `/api/config` spreads and every route reads, so an in-place mutation makes the merged schema instantly visible to /api/config, validate(), the relation identity layer, import coercion, and dup rules; the FILE is never written. Guard failures throw `{status:400}` and thrown ops are never logged (existing wrapper contract, proven in relations). SEED/DELTA COLLISION (seed changed under an old log): the replayed op hits its own duplicate-key guard, the replay loop's existing skip-and-warn absorbs it → **seed wins, deterministically**, zero special-casing. Deltas need no app-slug keying: the log is already per-store/per-warehouse; an in-memory fixture's deltas die with its process.

OPS + GUARDS (each block a 400 NAMING the reason): `schemaObjectAdd` — key slug `^[a-z][a-zA-Z0-9]*$`, unique vs objects, born with `{labels, icon?, primaryField {key,label} (type text)}` + empty rows bucket. `schemaFieldAdd` — object exists, field key unique + slug-valid, type in the supported set (composites + relation config incl. options-with-colors), select/multiselect require options, `unique` allowed (new field = no values). `schemaFieldUpdate` — allowed: label · options · isActive · width · scale · inverseLabel · unique; BLOCKED: `key` in patch (immutable), `type` when any live row holds a non-empty value (message names the row count; allowed at zero values), `isActive:false` on the primary field, `unique:true` when live duplicates exist (message NAMES the offending values). NOT EXPOSED at all in v1: object delete, field hard-delete. LOGGED_OPS += the three ops; replay proof = mock-warehouse journey (schema deltas + data writes interleaved, restart, byte-equal four-way snapshot — the rel-replay pattern).

SURFACE: `src/app/pages/SchemaPage.tsx` + one `pages.tsx` row `{key:"schema"}` — App.tsx untouched (its existing `features[key] !== false` filter hides the page; the ApiKeys self-guard pattern mirrors the server 403 for non-admin). env.mjs +2 pre-granted lines (`FEATURE_SCHEMA` zod + `schema: flagOn(...)`). Routes (flag off → 404; below admin → 403, `can`-style settings gate like apikeys): `POST /api/schema/objects` · `POST /api/schema/objects/:key/fields` · `PATCH /api/schema/objects/:key/fields/:fieldKey`. Page: objects list → per-object field table (key/type/label/flags/active) with inline label edit, options editor (value+label+color rows), retire/re-activate, add-field form (key, label, type select, options, unique), add-object form. After any commit: toast → `location.reload()` (v1 contract; /api/config re-serves merged). ObjectView toolbar "+ field" shortcut: SKIPPED — it needs an ObjectView+flag plumb while `#/p/schema` is one click away; declared cut, not silent.
